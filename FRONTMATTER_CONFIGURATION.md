# Frontmatter Configuration Guide

## For Content Teams

This guide shows you how to configure frontmatter validation for your content. Configure fields once, and the system automatically validates all content against your rules.

---

## Quick Start

**Add a new field to your module:**

```yaml
# config.yml (your module)
params:
  frontmatter:
    author:                           # Field name
      type: string                    # Field type
      required: true                  # Validation rule
      error_message: "Author required"
      placeholder_message: "Enter author name"
```

**That's it!** The system will:
- ✅ Validate this field in all content
- ✅ Show custom error messages when validation fails
- ✅ Provide placeholder guidance for content editors

---

## Where to Configure

### Option 1: Base Theme Config (For All Modules)

**File**: `config/_default/frontmatter/params.yaml`

Use this for fields that ALL modules should have:

```yaml
frontmatter:
  title:
    type: string
    required: true
    error_message: "Title is required"
    placeholder_message: "Enter page title"
```

### Option 2: Module Config (Module-Specific)

**File**: `config.yml` → `params.frontmatter`

Use this for fields specific to your module or to override base theme:

```yaml
params:
  frontmatter:
    category:                         # Add module-specific field
      type: string
      required: true
      options:
        - guides
        - reference

    title:                            # Override base theme field
      required: false                 # Change requirement
```

### How They Merge

```
Base Theme Fields
       ↓
  Deep Merge
       ↓
Module Fields
       ↓
Module Overrides Win
```

## Viewing the Final Schema

After configuration, view the unified schema at:

**File**: `data/presidium/frontmatter-schema.yaml` ⚠️ **READ-ONLY**

⚠️ **IMPORTANT**: This file is automatically generated. Do NOT edit manually!

**Protection**:
- Header warning in generated file
- Excluded from git (in `.gitignore`)
- Read-only file permissions (`chmod 444`)

**To regenerate**:
```bash
hugo --gc --config config.yml,dependencies.config.yml
cp public/frontmatter-schema.yaml data/presidium/frontmatter-schema.yaml
chmod 444 data/presidium/frontmatter-schema.yaml
```

**Example schema content**:

This file shows:

## Field Types

### Type 1: String

**Use for**: Text fields, short content, names, descriptions

**Required Attributes**:
```yaml
field_name:
  type: string                        # ✅ Required
  error_message: "Custom error"       # ✅ Required
  placeholder_message: "Help text"    # ✅ Required
```

**Optional Attributes**:
```yaml
  required: true                      # ⚙️ Optional (default: false)
  pattern: '^\w+$'                    # ⚙️ Optional (default: "")
  options: [opt1, opt2]               # ⚙️ Optional (default: [])
  display_rows: 3                     # ⚙️ Optional (default: 1)
```

**Example - Simple Text**:
```yaml
frontmatter:
  author_name:
    type: string
    required: true
    error_message: "Author name is required"
    placeholder_message: "Enter full name"
```

**Example - Text with Pattern**:
```yaml
frontmatter:
  slug:
    type: string
    required: true
    pattern: '^[a-z0-9-]+$'
    error_message: "Slug must be lowercase letters, numbers, and hyphens only"
    placeholder_message: "e.g., my-page-slug"
```

**Example - Text with Options**:
```yaml
frontmatter:
  status:
    type: string
    required: true
    options:
      - draft
      - review
      - published
    error_message: "Status must be draft, review, or published"
    placeholder_message: "Select status"
```

**Example - Multi-line Text**:
```yaml
frontmatter:
  description:
    type: string
    required: false
    display_rows: 5
    error_message: "Invalid description"
    placeholder_message: "Enter page description"
```

### Type 5: Object

**Use for**: Nested structures, grouped fields

**Required Attributes**:
```yaml
field_name:
  type: object                        # ✅ Required
  error_message: "Custom error"       # ✅ Required
  placeholder_message: "Help text"    # ✅ Required
  fields:                             # ✅ Required - nested fields (dict)
    nested_field:
      type: string
      error_message: "Nested error"
      placeholder_message: "Nested help"
```

**Optional Attributes**:
```yaml
  required: true                      # ⚙️ Optional (default: false)
  description: "Field purpose"        # ⚙️ Optional
```

**Example - Simple Object**:
```yaml
frontmatter:
  metadata:
    type: object
    required: true
    description: "Document metadata"
    error_message: "Metadata is required"
    placeholder_message: "Enter metadata"
    fields:
      author:
        type: string
        required: true
        error_message: "Author required"
        placeholder_message: "Enter author name"
      version:
        type: string
        required: true
        pattern: '^\d+\.\d+\.\d+$'
        error_message: "Version must be X.Y.Z format"
        placeholder_message: "e.g., 1.0.0"
```

**Example - Nested Objects**:
```yaml
frontmatter:
  page_config:
    type: object
    required: false
    error_message: "Invalid page config"
    placeholder_message: "Configure page"
    fields:
      metadata:
        type: object
        required: false
        error_message: "Invalid metadata"
        placeholder_message: "Enter metadata"
        fields:
          author:
            type: string
            required: true
            error_message: "Author required"
            placeholder_message: "Enter author"
      settings:
        type: object
        required: false
        error_message: "Invalid settings"
        placeholder_message: "Configure settings"
        fields:
          published:
            type: boolean
            required: false
            error_message: "Invalid publication status"
            placeholder_message: "Is published?"
```

```yaml
  - quickstart
  - tutorial

  name: "Jane Developer"
  email: "jane@example.com"
```

### Invalid Content - Missing Required

```yaml
---
# Missing 'title' (required field)
category: guides
---

Content here...
```

**Error**: `Field 'title': Title is required (required field is missing)`

---

### Invalid Content - Pattern Mismatch

```yaml
---
title: "Guide"
publish_date: "2025/01/27"  # Wrong format (should be YYYY-MM-DD)
---

Content here...
```

**Error**: `Field 'publish_date': value '2025/01/27' does not match pattern '^\d{4}-\d{2}-\d{2}$'`

---
```yaml
title: "Guide"
category: blog  # Not in allowed options [guides, tutorials, reference]
---

Content here...
```

**Error**: `Field 'category': value 'blog' is not in allowed options: [guides, tutorials, reference]`

## Configuration vs Generated Schema

### Configuration Files (What You Edit)

**Location**:
- `config/_default/frontmatter/params.yaml` (base theme)
- `config.yml` → `params.frontmatter` (your module)

**Purpose**: Define validation rules

**Example**:
```yaml
# You write this:
frontmatter:
  title:
    type: string
    required: true
    error_message: "Title required"
    placeholder_message: "Enter title"
```

---

### Generated Schema (Read-Only)

**Location**: `data/presidium/frontmatter-schema.yaml` ⚠️ **DO NOT EDIT**

**Purpose**: Complete schema with all defaults applied

**Protection Mechanisms**:
1. ⚠️ Header warning in generated file
2. 🚫 Excluded from git (`.gitignore`)
3. 🔒 Read-only file permissions (`chmod 444`)

**Example**:
```yaml
################################################################################
# ⚠️  DO NOT EDIT THIS FILE MANUALLY
################################################################################
# [Full header warning with instructions]
################################################################################

# System generates this:
title:
  type: string
  required: true
  error_message: "Title required"
  placeholder_message: "Enter title"
  options: []           # ← Default added
  pattern: ""           # ← Default added
  display_rows: 1       # ← Default added
```

**Key Differences**:

| Aspect | Configuration Files | Generated Schema |
|--------|-------------------|------------------|
| Edit? | ✅ YES - You edit these | ❌ NO - Read only |
| Purpose | Define rules | Complete schema for validation |
| Defaults | Omit optional fields | All defaults applied |
| Location | config/ directory | data/ directory |
| Version control | ✅ Committed to git | ❌ Excluded (.gitignore) |
| Permissions | Read/write | Read-only (chmod 444) |

## Troubleshooting

### Problem: "Field not validated"

**Symptom**: Content has invalid data but no error shown

**Check**: Did you configure the field?

```yaml
# config.yml
params:
  frontmatter:
    your_field:  # ← Must be configured
      type: string
      required: true
      error_message: "Error message"
      placeholder_message: "Help text"
```

---

### Problem: "Module config not working"

**Symptom**: Module fields not appearing in validation

**Check**: Config structure

```yaml
# ✅ Correct
params:
  frontmatter:
    field_name:
      type: string

# ❌ Wrong (missing 'params' wrapper)
frontmatter:
  field_name:
    type: string
```

---

### Problem: "Can't see my changes"

**Symptom**: Updated config but validation hasn't changed

**Check**: Generated schema updated?

```bash
# View generated schema
cat data/presidium/frontmatter-schema.yaml

# If not updated, regenerate schema
hugo --gc --config config.yml,dependencies.config.yml
cp public/frontmatter-schema.yaml data/presidium/frontmatter-schema.yaml
chmod 444 data/presidium/frontmatter-schema.yaml
```

---

### Problem: "Accidentally edited generated schema"

**Symptom**: Made changes to `data/presidium/frontmatter-schema.yaml` but they disappeared

**Reason**: This file is auto-generated and overwritten on regeneration

**Fix**:
1. Undo changes to generated schema
2. Make edits in config files instead:
   - `config/_default/frontmatter/params.yaml` (base theme)
   - `config.yml` → `params.frontmatter` (your module)
3. Regenerate schema from config

**Prevention**: File has read-only permissions (`chmod 444`) to prevent accidental edits

---

### Problem: "Pattern not working"

**Symptom**: Pattern validation not triggering

**Check**: Pattern syntax (use regex)

```yaml
# ✅ Correct (regex pattern)
pattern: '^\d{4}-\d{2}-\d{2}$'

# ❌ Wrong (not regex)
pattern: 'YYYY-MM-DD'
```

```yaml

field_name:
```

```yaml
field_name:
```

```yaml
field_name:
```

```yaml
field_name:
  type: object
```
